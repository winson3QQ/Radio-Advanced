情境1：CTCSS 亞音頻

現場困境

地震後 5 支隊伍共用唯一申請到的頻率 145.500MHz，所有隊伍同時搶頻講話，指揮所只聽到雜訊，救援協調完全癱瘓。

RF 理論：CTCSS 的本質

關鍵概念：CTCSS 不是新頻率，而是訊號疊加

一個對講機發出的 FM 信號，同時包含三層：

  • 第 1 層（語音）：300–3000Hz，人耳可清晰聽到
  • 第 2 層（CTCSS 亞音）：67–250Hz 之一。這個頻段人耳其實聽得到，但會被收訊端約 300Hz 的高通濾波與通訊音頻的低頻滾降濾掉——所以「聽不太到」是被濾掉，不是耳朵聽不見
  • 第 3 層（FM 載波）：145.500MHz，無線電的「傳輸馬路」

SDR 驗證實驗
工具 — GQRX + RTL-SDR + 2 支對講機

步驟1：接收單一 CTCSS 信號

  • 對講機A設 CTCSS = 67.0Hz，在 145.500MHz 發話
  • GQRX：接收模式 NFM，帶寬 12.5kHz
  • 點選 Audio → 看頻率響應，在 67Hz 附近應出現尖峰

步驟2：比較不同 CTCSS 頻率

  • 對講機A換成 CTCSS = 88.5Hz，重複觀測
  • 尖峰位置從 67Hz 移到 88.5Hz → 直接驗證 CTCSS 嵌入

步驟3：兩隊「輪流」發話（正確用法）

  • 對講機A：CTCSS 67Hz；對講機B：CTCSS 88.5Hz
  • A 先講完、B 再講 → 設接收 CTCSS=67Hz 只聽到 A；設 88.5Hz 只聽到 B ✓
  • 這證明 CTCSS 能讓你「選聽其中一隊」，前提是兩隊不同時佔用頻率

步驟3b：兩隊「同時」發話（觀察碰撞）

  • 讓 A、B 在 145.500MHz 同一瞬間按下發話鍵
  • FM 捕獲效應：接收機只鎖定較強的一方，較弱者被壓掉；功率相近時則是一團亂碼
  • 就算把接收 CTCSS 設成弱方的碼，也救不回來——射頻已在空中相撞，CTCSS 只是接收端的篩選
  • 結論：CTCSS 不能讓多隊「同時」共存，只能搭配輪流發話（通訊規程）分時共用同一頻率

步驟4（GNU Radio 進階）：抽出 CTCSS 信號

  • 流程：RTL-SDR Source → FM Demod → Low Pass Filter(300Hz) → Audio Sink
  • 低通濾波器截止 300Hz → 聽到 CTCSS 的嗡嗡聲
  • 改成帶通濾波器：65–70Hz → 只留下 67Hz CTCSS
  • FFT Sink 顯示清晰的 67Hz 峰值 → 理論完全驗證
切到 88.5Hz 時尖峰會往右移到 88.5 的位置——用 FFT 就能確認 CTCSS 設定是否正確。
現場應急應用
隊伍數量CTCSS 分配策略指揮所設定
2–5 支標準 38 碼，各隊一個無 CTCSS（接收全部）
6–10 支選擇間隔較大的碼（避免互調）無 CTCSS
11+ 支考慮分配第二個頻率無 CTCSS
注意（事實）CTCSS 不提供通訊保密！只要不設 CTCSS 的接收機都能聽到全部通話。
記憶方法CTCSS 是「靜噪鑰匙」而非「加密鑰匙」——它決定接收喇叭開不開，不是頻率隔離。
容易誤會CTCSS 不能讓多隊「同時」在同一頻率講話互不干擾。同時發射會因 FM 捕獲效應相撞（強者勝、弱者被吃掉）。它真正的價值是搭配輪流發話，讓多隊「分時」共用一個頻率，並各自只聽自己關心的那組。

情境2：RF 增益與接收靈敏度

現場困境

山區搜救隊距指揮所 5km，信號微弱，調到最大音量仍聽不清楚。指揮官問：「除了大聲，還有什麼辦法？」

RF 理論：靈敏度與增益的關係

接收機的本質問題：信號和雜訊都會被放大

  • 信號弱時，增益↑：信號和底噪一起放大，但信號相對提升 → 清晰度改善
  • 信號強時，增益↑：信號已足夠，只是加更多噪音 → 清晰度反而降低
  • 增益過高：接收機前端過載（Overload），產生非線性失真 → 聽到嗡嗡聲或嘯叫
SDR 驗證實驗
工具 — GQRX + RTL-SDR + 對講機（500m 外發話）

步驟1：建立基準

  • GQRX 設定：NFM 模式，頻寬 12.5kHz，目標頻率 145.500MHz
  • RF 增益設為 0dB，記錄 FFT 信號高度和底噪水平

步驟2：增益爬升測試

  • 每次增加 5dB，記錄：FFT 峰值高度、底噪水平、SNR、語音清晰度（1-5 分）
  • 重複：0→5→10→15→20→25→30→35→40→49.6dB

步驟3：繪製 SNR 曲線

  • 橫軸：增益（dB）；縱軸：SNR（dB）
  • 找出「最佳增益點」（SNR 最大值對應的增益）

步驟4：重複不同距離

  • 對講機移到 100m、300m、1km、2km 重複測試
  • 發現：距離越遠，最佳增益點越高 → 形成規律
RF 增益FFT 峰值語音
0 dB幾乎沒入底噪無
20 dB小峰冒出微弱
35 dB（最佳）峰高、SNR 最大清晰
49 dB（過載）底噪一起抬高噪聲蓋過
增益不是越高越好——目標是讓「峰值−底噪」差值（SNR）最大的那個點。
現場應急應用
距離/環境建議起始增益調整方向警告訊號
近距離市區（<300m）10–15dB信號好不需再增嗡聲=過載→降低
中距離（300m–1km）20–30dB視 FFT 峰值調整背景嘶聲=底噪高
山區/遠距（1km+）30–40dB找 SNR 最大點仍無信號=天線問題
城市大樓（多反射）15–25dB先降後調過多多路徑=換位置
記憶方法「最大增益 ≠ 最好接收」。每個環境都有一個最佳增益點，用 FFT 的 SNR 就能找到。

情境3：頻寬與頻譜效率

現場困境

中繼台停電後隊伍改用直聯，但通話距離從 5km 驟降到不到 2km。查對講機設定，發現部分隊伍設的是 25kHz 寬頻。

RF 理論：頻寬與能量密度

核心物理原理：

  • 相同發射功率，帶寬越窄 → 能量越集中 → 每 Hz 的功率密度越高
  • 能量密度越高 → 接收機更容易從雜訊中識別信號
  • 識別容易 → 在更遠距離仍能清晰通話
SDR 驗證實驗
工具 — GQRX + RTL-SDR + 2 支不同頻寬設定的對講機

步驟1：測量 FFT 波形寬度

  • 對講機A設 25kHz 模式在 145.500MHz 發話
  • GQRX：看 FFT 峰值的 -3dB（半功率）寬度，應約 25kHz
  • 對講機B設 12.5kHz 模式發話，重複測量
  • 視覺對比：25kHz 的山型明顯更寬

步驟2：SNR 比較測試（相同距離）

  • 兩支對講機在相同地點、相同功率，距離 500m 發話
  • 分別用 GQRX 測量接收到的 SNR（信號高度 - 底噪高度）
  • 預期：12.5kHz 的 SNR 比 25kHz 高約 3dB

步驟3：通話距離測試（關鍵實驗）

  • 兩支對講機逐步後退，每 50m 記錄通話清晰度（1-5 分）
  • 記錄「通話失去清晰」的距離（清晰度降至 3 分以下）
  • 預期：12.5kHz 清晰距離比 25kHz 遠約 30-40%（自由空間量級；實際地形通常更小）
能量集中 → 功率密度更高 → 相同功率下信號更清晰。
錯誤示範（設定錯誤→干擾）

一隊把對講機誤設成 25kHz 寬頻（或麥克風增益開到爆、過度頻偏），訊號的裙邊往外溢，咬到相鄰 12.5kHz 頻道正在通聯的隊伍。

教官瀑布圖示範：把寬頻那台和窄頻那台擺在相鄰頻道，瀑布圖上立刻看到寬頻的訊號柱「胖」到壓進隔壁那條——這就是鄰頻干擾的長相。處置：要求該隊改回 12.5kHz、降低麥克風增益。

現場應急應用
情況推薦頻寬原因
有中繼台支援12.5kHz（仍建議）現代標準，節省電力
直聯（無中繼）12.5kHz（強制）能量密度高，距離更遠
與舊式對講機配合視對方設定需確認對方設定
強電池使用時間優先12.5kHz相同覆蓋所需功率更低
記憶方法12.5kHz 窄頻是「用更少能量走更遠的路」。災害時節省電力和擴大覆蓋都靠它。

情境4：CTCSS 層級管理

現場困境

大樓倒塌 40 人被困不同樓層，1 台對講機需協調 5 支搜救隊，結果 40 人搶著發話，指揮所聽到一片混亂。

RF 理論：頻率共享的信道容量

（CTCSS 的本質與「選聽」原理已在情境1 講過，這裡聚焦多隊分層的系統設計，不重複推導。）

Shannon 容量公式（推論方向）：

  • 一個頻道（145.500MHz）在同一時刻只能承載 1 個信號，不能同時多人發話
  • CTCSS 是「接收側的過濾器」，不能解決同時發話的物理衝突
  • 真正的解法：用通訊規程（SOP）管理發話順序
  • CTCSS 的作用：讓每層搜救隊只聽到自己層的指令，不被其他層干擾
系統設計原則
層級化通訊模型： 主頻率：145.500MHz（唯一申請） CTCSS碼分配： 指揮所：無CTCSS（聽全部） ├─ 1–2樓隊：CTCSS 67.0Hz ├─ 3–4樓隊：CTCSS 88.5Hz ├─ 5–6樓隊：CTCSS 110.9Hz └─ 7–8樓隊：CTCSS 131.8Hz 發話流程（SOP）： 指揮所 → 廣播（設無CTCSS發話：所有人都收到） 各樓層隊 → 只發自己的CTCSS（其他層聽不到） 優點： ・每層只聽指揮所的廣播 + 自己的確認 ・其他層的日常報告不干擾 ・指揮所統一掌握所有資訊 CTCSS碼選擇建議（避免解碼誤判）： 選擇頻率差距大、且非整數倍關係的碼 例：67.0, 88.5, 110.9, 131.8 → 間隔夠大 ✓ 避免：67.0, 71.9, 74.4 → 太接近，解碼器容易誤判 避免諧波關係（某碼 ≈ 另一碼的整數倍）造成誤觸發 ※這是「解碼誤判／諧波」問題，不是射頻互調(IMD)
SDR 驗證實驗
工具 — GQRX 瀑布圖 + 4 支對講機（分別設不同 CTCSS）

步驟1：建立層級通訊系統

  • 對講機1（指揮所）：145.500MHz，無 CTCSS
  • 對講機2（1-2樓）：145.500MHz，CTCSS 67.0Hz
  • 對講機3（3-4樓）：145.500MHz，CTCSS 88.5Hz
  • 對講機4（5-6樓）：145.500MHz，CTCSS 110.9Hz

步驟2：瀑布圖監控（最具視覺效果）

  • GQRX 瀑布圖顯示在大螢幕
  • 各樓層輪流報告：觀察瀑布圖中通話時間軸
  • 指揮所發廣播：瀑布圖顯示廣播信號（強）
  • 各層確認：瀑布圖顯示輕微的回覆信號

步驟3：驗證 CTCSS 選聽效果（各隊輪流報告）

  • 對講機設接收 CTCSS=67Hz → 只聽到 1-2 樓的報告
  • 其他樓層通話時 → 靜噪不開，不干擾
  • GQRX「音頻頻譜」顯示：切換 CTCSS 接收值 → 峰值位置改變

情境5：干擾檢測與定位

現場困境

救援 12 小時後，指揮所突然聽到陌生聲音插進來：「天氣很好，今天適合釣魚」。通訊協調被打斷，無法確認干擾來源。

RF 理論：干擾的類型與物理特徵

FM 通訊中常見干擾類型（事實）：

  • 同頻干擾（Co-channel）：兩個信號在同一頻率上，FM 的「捕捉效應」讓強者獲勝，但若強度相近則雙方都失真
  • 鄰頻干擾（Adjacent-channel）：相鄰頻率的強信號「洩漏」到目標頻率
  • 互調干擾（IMD）：兩個強信號在接收機非線性電路中混頻，產生新的虛假頻率
SDR 驗證實驗
工具 — GQRX 掃頻模式 + RTL-SDR

步驟1：定位干擾頻率

  • GQRX 設定：掃描模式，範圍 144.0–147.0MHz
  • 觀察 FFT：在目標頻率以外找異常峰值
  • 紀錄干擾頻率（例：145.510MHz）

步驟2：判斷干擾類型

  • 干擾在目標頻率正上方 → 同頻干擾
  • 干擾在目標頻率旁邊 ±12.5kHz → 鄰頻干擾
  • 多個異常峰值，等間隔分布 → 互調干擾

步驟3：信號強度三角定位

  • 在不同位置（A、B、C 點）測量干擾信號的 RSSI
  • A 點（東邊）：RSSI = S7
  • B 點（北邊）：RSSI = S4
  • C 點（西邊）：RSSI = S2
  • 結論：干擾源靠近 A 點（東邊），最強信號方向即為來源方向

※ 純 RSSI 三角定位很粗略。要準確測向（DF）需要定向天線（八木／環形）並實際移動位置——這比較適合當外場分組練習，不是教官桌上的瀑布圖示範。

步驟4：驗證換頻率的效果

  • 掃描 145.0–146.0MHz 全段，找無信號的乾淨頻率
  • 隊伍換到新頻率後，用 GQRX 確認無干擾
  • 對比：換前 FFT 雜亂 vs 換後 FFT 乾淨
干擾更強 → FM 捕捉效應讓它壓過我方信號。三角定位推算方向、快速換頻是最有效處置。
錯誤示範（設定錯誤→干擾）

把接收增益（或外接放大器）開到爆，接收機前端過載進入非線性區，兩個強台在機內混頻產生 2f₁−f₂、2f₂−f₁ 的互調(IMD)鬼影，憑空多出根本不存在的「干擾台」。很多人會誤把這當成真的有人在干擾。

教官瀑布圖示範：把增益拉到過載，瀑布圖上會冒出一排等間隔的鬼影峰；把增益調回正常，它們同時消失——證明那是「自己機器產生的」假訊號，不是空中真有干擾。處置：先降增益／加裝衰減器再判斷。

情境6：中繼台設計與雙工原理

現場困境

文山區與信義區搜救隊需要協調，直線距離 3km 但中間有山丘遮擋，直聯完全無法通話。

RF 理論：中繼台的物理原理

為什麼需要中繼台：地形遮擋 = 菲涅耳區阻礙

  • 無線電信號在「視距（Line of Sight）」外顯著衰減
  • 山丘或建築物遮擋菲涅耳區 → 信號強度損失 10–30dB
  • 中繼台放在高點 → 與兩個端點都保持視距 → 信號轉發

雙工的必要性（重要！）

  • 中繼台需同時接收和發射
  • 如果收發頻率相同：自己的發射信號進入自己的接收機 → 自激 → 嘯叫
  • 解法：收發使用不同頻率（頻率對）
SDR 驗證實驗
工具 — GQRX 雙頻監聽 + RTL-SDR

步驟1：設置中繼台通訊系統

  • 文山隊對講機：發 145.500，收 146.100MHz（設 offset +600kHz）
  • 信義隊對講機：發 145.500，收 146.100MHz（同上）
  • 中繼台：收 145.500，發 146.100MHz（自動轉發）

步驟2：GQRX 雙頻監聽驗證

  • GQRX 打開兩個接收視窗：145.500 和 146.100MHz
  • 文山隊發話 → 145.500 FFT 出現峰值 → 146.100 FFT 稍後出現（中繼延遲）
  • 兩個 FFT 圖的時間差即為中繼台延遲（通常 <100ms）

步驟3：驗證中繼台健康狀況

  • 文山隊每分鐘發一次信標（短信號）
  • GQRX 瀑布圖：145.500 和 146.100 各出現規律信號條紋
  • 若 146.100 無信號 → 中繼台故障 ← 立即可知

步驟4：比較直聯 vs 中繼信號強度

  • 直聯（地形遮擋）：GQRX 幾乎無法接收到信號
  • 透過中繼台：GQRX 接收到明顯峰值（S7 以上）
  • 差距量化：中繼帶來的信號強度提升（dB 數值）

CTCSS 在中繼台的應用

中繼台架在高處、收訊靈敏、涵蓋極廣。若只靠單純的載波靜噪（Carrier Squelch），很容易被遠端雜訊、非法電波或隔壁縣市的同頻訊號觸發，導致中繼台無故啟動、持續發射（俗稱空轉、拉尾巴）。CTCSS 亞音頻就是中繼台的「鑰匙＋門禁」——核心目的有二：防止干擾與控制轉播權限。

發話時，對講機會在語音下方疊加一個聽不太到的低頻連續音（67.0–250.3 Hz）。中繼台內部 CPU 濾波解碼後，只有 Tone 碼完全吻合才會啟動發射機轉播；不帶碼或帶錯碼的訊號一律被過濾、不予理會。

應用一　上行鎖定：控制轉播啟動（最常見）

中繼台在接收端（上行 Uplink）設一組 CTCSS（如 88.5 Hz）。使用者按 PTT 發話時帶上此 Tone，碼吻合 → 中繼台啟動轉播；沒帶碼的雜訊與同頻干擾全被擋掉。

應用二　下行鎖定：減少使用端收訊干擾

中繼台發射端（下行 Downlink）可選擇要不要也送 Tone：

  • 只開上行：使用者只需開「發射 Tone（T）」即可觸發中繼，轉播出來的訊號不帶 Tone。
  • 雙向開啟：中繼台轉播時也疊加 Tone，使用者再開「接收 Tone（R）」→ 喇叭只在中繼台真正轉播時才打開，徹底隔絕環境雜訊與其他同頻對講機的直通干擾。

應用三　同頻不同 Tone：頻率重複利用

頻率資源緊張時，有地理阻隔的兩隊可共用同一中繼頻率：A 隊中繼台設 100.0 Hz、B 隊設 141.3 Hz。收發頻率一模一樣，但內碼不同 → A 隊說話絕不會觸發 B 隊的中繼台，反之亦然，大幅提升頻道利用率。

狀態落在中繼台接收頻率？帶正確 CTCSS？中繼台反應
正常通話是是（正確）啟動轉播，擴大通訊範圍
同頻干擾是否／錯誤拒絕轉播，保持靜音、不干擾
環境雜訊否（落於盲區）否拒絕轉播，完全屏蔽
三種情況只有「頻率對 + Tone 對」才會啟動轉播——這就是 CTCSS 既擋干擾、又能讓多隊共用同頻的關鍵。
記憶方法使用中繼台要設三樣：頻率（如 145.500）＋偏移（如 +600kHz）＋發射 CTCSS（Tone / T）。少設 CTCSS，按了發話鍵中繼台也不會動。
常見誤解中繼台的 CTCSS 是「存取碼（開門用）」，和情境 1 用 CTCSS「選聽分隊」目的不同——但原理一樣：都是用亞音頻控制「要不要打開」。它一樣不是加密，沒設 CTCSS 的人仍聽得到中繼台轉發的內容。
生活比喻中繼台像一台「公用擴音器」，但要先刷對門禁卡（CTCSS）它才幫你廣播；刷錯卡或不刷，喊破喉嚨它也不理你。
錯誤示範（設定錯誤→干擾／打不開）

偏移方向設反（該 + 卻設 −），或忘了開發射 Tone，是最常見兩種錯：

  • 忘開 Tone：按 PTT 中繼台不動，常被誤判成「中繼台壞了」，其實是門禁碼沒帶。
  • 偏移設反：對講機直接發在中繼台的輸出頻率（下行）上，等於拿石頭砸自家喇叭——壓掉所有人正在收聽的轉播訊號。

教官瀑布圖示範：同時監看輸入（145.500）與輸出（146.100）兩條頻率。帶錯碼／沒帶碼 → 輸出那條不出現；偏移設反 → 輸出頻率上冒出不該有的直發訊號蓋住轉播。Tone、偏移一改對，輸出柱立刻正常——因果一目了然。
